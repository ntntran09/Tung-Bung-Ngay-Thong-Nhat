# CO_GANH

## Purpose

`CO_GANH/` contains the Co Ganh mini-game: bot selection, board setup, player moves, bot moves, capture rules, and win/loss UI.

## Related Docs

- [../architecture.md](../architecture.md): how this module is launched from and returns to the hub.
- [MAIN.md](MAIN.md): shared pause UI and hub-owned launch behavior.
- [../runbooks/manual-smoke-test.md](../runbooks/manual-smoke-test.md): Co Ganh smoke checks.

## What This Module Is Responsible For

- displaying bot/avatar selection
- spawning the board pieces
- showing legal move hints
- applying player and bot moves
- handling `ganh` and `vay` conversion rules
- showing win/loss panels
- returning to the hub from end states or pause UI

## What This Module Is Not Responsible For

- hub dialogue that launches the game
- backend API communication
- persistent score or progression
- project-wide input mapping

## Structure

- `scenes/main.tscn`: main Co Ganh scene.
- `scenes/Piece.tscn`: spawned piece scene.
- `scripts/board.gd`: main board and game controller.
- `scripts/bot_select_layer.gd`: bot selection UI and `bot_selected` signal.
- `scripts/bot_logic.gd`: bot move selection, minimax, and alpha-beta helper logic.
- `scripts/piece.gd`, `scripts/pieces.gd`: piece-specific helpers.
- `scripts/highlight.gd`: UI hover/highlight behavior.
- `scripts/point.gd`: point display helper.
- `assets/`: sprites, sounds, and music for the module.

## Entry Points

- From the hub: `res://CO_GANH/scenes/main.tscn`
- Direct debug entry: `CO_GANH/scenes/main.tscn`

## Inputs And Outputs

### Inputs

- mouse input for bot selection and board interaction
- board state generated by `board.gd`
- selected bot level from `bot_select_layer.gd`

### Outputs

- runtime board state in `board.gd`
- win or loss panel visibility
- return to `MAIN/scenes/main.tscn`

## Important Runtime Assumptions

### Board setup is code-driven

`board.gd` creates the valid move graph, spawns initial pieces, and maps grid coordinates to pixel positions. Scene visuals and code assumptions must stay aligned.

### Bot selection gates gameplay visibility

On `_ready()`, `board.gd` hides the board, pieces, movement hints, player UI, and point UI. Gameplay becomes visible only after `BotSelectLayer.bot_selected` is emitted.

### Piece side names are code strings

The current code uses:

- `Lua`
- `Nuoc`

Renaming these string values requires updates across board and bot logic.

### Shared pause UI is imported from `MAIN`

`CO_GANH/scenes/main.tscn` instances `MAIN/scenes/ui/game_control.tscn` for the shared pause menu. Keep that packed-scene path instead of embedding `MAIN/script/game_control.gd` and pause assets directly.

## Common Workflows

### Change bot difficulty

Read first:

- `scripts/bot_select_layer.gd`
- `scripts/bot_logic.gd`
- `scripts/board.gd`

Then re-test bot selection, first player move, and first bot response.

### Change board rules

Read first:

- `scripts/board.gd`
- `scripts/bot_logic.gd`

Then re-test:

- legal move hints
- `ganh`
- `vay`
- win/loss detection
- reset/replay flow

### Change end-state UI

Read first:

- `scenes/main.tscn`
- `scripts/board.gd`

Then re-test both win and loss panels and hub return buttons.

## Artifacts

- runtime-only board state
- no visible save file
- module-local art and audio under `CO_GANH/assets/`

## Change Impact

- editing `board.gd` affects nearly all gameplay
- editing `bot_logic.gd` can make the bot choose illegal or empty moves
- editing node names in `scenes/main.tscn` can break many `$...` paths in `board.gd`
- editing shared pause UI can affect this module and other mini-games

## What To Test After Changes

1. Launch `CO_GANH/scenes/main.tscn`.
2. Select a bot level.
3. Confirm the board appears after confirmation.
4. Select one player piece and click a valid move hint.
5. Confirm the bot responds.
6. Trigger restart or replay if touched.
7. Return to `MAIN/scenes/main.tscn`.
